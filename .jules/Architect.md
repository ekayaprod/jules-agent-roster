## Architect — Structural reinforcement of JulesManager
**Learning:** Monolith components managing complex state machines (e.g. JulesManager) can be safely split into distinct domain modules (JulesModals, TerminalPolling). However, when doing so in a Vanilla JS application without a bundler, ensure that the newly split dependencies are correctly loaded in the HTML and explicitly attached to the global scope during jsdom test setups to prevent undefined reference errors.
**Action:** Always map back test suite failures after an extraction to the underlying module boundaries and ensure proper DI instantiation inside the parent constructor.

**Learning:** Massive monoliths orchestrated entirely by event delegation patterns can have their event listener payloads cleanly extracted into pure static `EventBinder` classes by injecting the `app` instance boundary, reducing line weight and visual clutter without altering behavior.
**Action:** Re-bind listeners via dependency injection (app context) and ensure optional chaining is safe or mapped correctly for native evaluations.
## Architect — Structural reinforcement of JulesTerminal
**Learning:** I encountered a massive monolithic file in `js/UI/JulesTerminal/JulesTerminal.js` containing both modal logic and polling logic. By splitting these responsibilities into `JulesModals.js` and `TerminalPolling.js`, I significantly improved modularity. However, since I am in a Vanilla JS environment without a bundler, I needed to make sure they were properly instantiated in the `JulesTerminal` constructor (`this.modals = new JulesModals(this); this.polling = new TerminalPolling(this);`) and that the global `index.html` included the new script tags correctly.
**Action:** Splitting a class in Vanilla JS requires exposing the new classes globally for tests (using `global.ClassName = ClassName`), linking them appropriately in `index.html`, and correctly delegating all previous method calls to the newly instantiated class fields to maintain functionality without breaking the contract with tests.
## Architect — Structural reinforcement of FusionLab
**Learning:** Found a structural monolith in \`js/Features/Fusion/FusionLab.js\` taking over 500 lines. The file handled not only fusion orchestration logic, but also detailed DOM event binding (like \`resetLabBtn.addEventListener\`) and manual DOM manipulation (toggling classes for slot icons).
**Action:** Decoupled the event bindings into a \`FusionLabEvents\` class and the view logic into a \`FusionLabRenderer\` class, stored within a dedicated \`js/UI/FusionLab\` domain directory. Hooked them up into the HTML and \`benchmark.js\` suite, preserving logic while distributing weight off the core orchestrator.
